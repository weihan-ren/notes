# PPO（Proximal Policy Optimization）详解

## 摘要

PPO 是当前深度强化学习领域最广泛使用的策略梯度算法。由 OpenAI 的 Schulman et al. (2017) 提出，它通过简洁的"裁剪"机制实现了稳定的策略更新，既避免了 TRPO 的计算复杂性，又保持了信任域方法的优势。在 LLM 领域，PPO 是 RLHF（InstructGPT/ChatGPT）的核心 RL 引擎。

---

## 1. 为什么需要 PPO —— 从 Policy Gradient 到信任域

### 1.1 策略梯度的问题

策略梯度方法的核心更新规则：
```
θ_new = θ_old + α · ∇J(θ)
```

**问题**：学习率 α 极难选择：
- α 太小 → 训练极慢
- α 太大 → 一步更新可能摧毁整个策略（策略崩溃）

在深度 RL 中，参数空间的微小变化可能导致策略分布的剧烈改变——因为神经网络的参数空间和策略空间之间的关系高度非线性。

### 1.2 TRPO 的解决方案

**TRPO（Trust Region Policy Optimization）**（Schulman et al., 2015）提出了"信任域"的概念：

- 每次更新时，保证新策略与旧策略之间的 KL 散度不超过一个阈值 δ
- 约束优化问题：`max L(θ) s.t. D_KL(π_old || π_new) ≤ δ`
- 使用共轭梯度 + 线搜索来求解

**TRPO 的问题**：
- 需要计算 Hessian 向量积
- 实现极其复杂（数百行代码）
- 与 dropout、参数共享等常用技术不兼容

---

## 2. PPO 的核心思想

PPO 的目标和 TRPO 一样——限制每次更新的幅度。但 PPO 用一种极其简单的方法实现：

> 与其用复杂的二阶约束，不如直接"裁剪"概率比，防止一次更新走太远。

PPO 有两个主要变体：
1. **PPO-Clip**（最常用，也是本文重点）
2. **PPO-Penalty**（自动调整 KL 惩罚系数）

### 2.1 PPO-Clip 目标函数

```
L_CLIP(θ) = E_t[ min( r_t(θ) · A_t , clip(r_t(θ), 1-ε, 1+ε) · A_t ) ]
```

其中：
- `r_t(θ) = π_θ(a_t|s_t) / π_θold(a_t|s_t)` — 新旧策略的概率比
- `A_t` — 优势估计（这个动作比基准好多少）
- `ε` — 裁剪范围（通常 0.1 ~ 0.2）

### 2.2 裁剪的直觉

| 情况 | r_t 的值 | 含义 | PPO 的处理 |
|------|----------|------|-----------|
| A_t > 0（好动作） | r_t >> 1+ε | 策略想大幅增加该动作概率 | 裁剪到 1+ε，阻止过度自信 |
| A_t > 0（好动作） | r_t < 1+ε | 策略想适度增加 | 正常更新 |
| A_t < 0（坏动作） | r_t << 1-ε | 策略想大幅降低该动作概率 | 裁剪到 1-ε，阻止过度惩罚 |
| A_t < 0（坏动作） | r_t > 1-ε | 策略想适度降低 | 正常更新 |

**核心直觉**：
- 当策略想大幅改好动作时 → "慢慢来，别太激进"
- 当策略想大幅改坏动作时 → "也别反应过度"
- `min()` 确保裁剪是悲观的——只在裁剪降低目标时才裁剪

## 3. GAE（广义优势估计）

### 3.1 为什么需要 GAE

在 PPO 目标中，`A_t` 是优势估计。问题是如何估计 `A_t`：
- 用实际回报 → 无偏但方差大
- 用 TD 误差 → 有偏但方差小
- 用 n 步 TD → 中间选项，但 n 选多少？

### 3.2 GAE 的解决方案

**GAE（Generalized Advantage Estimation）**（Schulman et al., 2016）通过指数加权平均解决了这个权衡：

```
A_GAE = Σ_l (γλ)^l · δ_{t+l}
```

其中 `δ_t = r_t + γ·V(s_{t+1}) - V(s_t)` 是 TD 误差。

- `γ`：折扣因子（通常 0.99）
- `λ`：GAE 参数（0 ~ 1），控制偏差-方差的权衡
  - `λ=0`：只用 TD(0)（高偏差、低方差）
  - `λ=1`：用 Monte Carlo 回报（低偏差、高方差）
  - 典型值：`λ=0.95`

**GAE 的优势**：
- 不需要选择 n 步的具体值
- 平滑地在偏差和方差之间进行权衡
- 与 PPO 配合极好

## 4. PPO 的训练架构

### 4.1 Actor-Critic 架构

```
┌──────────────────────────────┐
│         环境 (Environment)    │
│                               │
│  状态 s_t ──→ Actor (策略) │
│                    ↓         │
│              动作 a_t         │
│                    ↓         │
│  状态 s_t ──→ Critic (价值) │
│                    ↓         │
│              V(s_t) 估计     │
└──────────────────────────────┘
```

- **Actor（策略网络）**：输出动作概率分布 `π(a|s)`
- **Critic（价值网络）**：输出状态价值估计 `V(s)`
- 两个网络可以共享底层特征提取器

### 4.2 PPO 完整训练循环

```
1. 用当前策略 π_θold 收集一批经验 {s_t, a_t, r_t, s_{t+1}}
2. 用 GAE 计算每个时间步的优势 A_t
3. 计算回报 R_t = A_t + V_old(s_t)
4. 对 K 个 epoch（通常 K=3~10）：
   a. 用小批量数据计算 PPO-Clip 损失 L_CLIP(θ)
   b. 计算价值函数损失 L_VF = (V(s_t) - R_t)²
   c. 可选：加熵正则化 L_ENT = -H(π)
   d. 总损失 = L_CLIP - c1·L_VF + c2·L_ENT
   e. 用 Adam 更新参数
5. θ_old ← θ（更新旧策略）
6. 清空经验缓冲区，回到步骤 1
```

## 5. PPO 的关键超参数

| 超参数 | 典型值 | 作用 | 调参建议 |
|--------|--------|------|----------|
| **裁剪 ε** | 0.1 ~ 0.2 | 控制更新幅度 | 越大学习越快但越不稳定 |
| **GAE λ** | 0.95 | 偏差-方差权衡 | 通常在 0.9~0.98 之间 |
| **折扣 γ** | 0.99 | 未来奖励的权重 | 长期任务用大值 |
| **更新 epoch K** | 3 ~ 10 | 每批数据用几次 | 简单任务 3-5，复杂任务 10 |
| **小批量大小** | 64 ~ 256 | 每次梯度更新用多少样本 | 大 batch 更稳定 |
| **熵系数 c2** | 0.01 | 鼓励探索 | 太小则探索不足 |
| **学习率** | 3e-4 | Adam 学习率 | 需要 warmup 和衰减 |

## 6. PPO 在 LLM 中的应用

### 6.1 RLHF 中的 PPO

在 RLHF 中，PPO 的状态、动作和奖励被重新定义：

| 传统 PPO 概念 | RLHF 中的对应 |
|--------------|--------------|
| 状态 s_t | 当前上下文（提示 + 已生成的 token） |
| 动作 a_t | 生成下一个 token |
| 策略 π(a|s) | LLM 的 token 概率分布 |
| 奖励 r_t | RM 打分 + KL 惩罚 |
| 轨迹 | 完整的多 token 生成序列 |

### 6.2 RLHF 中的特殊处理

1. **KL 惩罚融入奖励**：
   ```
   r_total = r_RM - β · log(π_RL(token) / π_SFT(token))
   ```
   逐 token 计算，不是轨迹级别。

2. **价值函数共享**：
   通常将价值头和策略头附加在同一个 transformer 骨干网络上。

3. **信用分配**：
   在文本生成中，只有最后一个 token 后才有 RM 打分。中间 token 的奖励全靠 KL 惩罚和 GAE 的时序传播。

4. **大模型挑战**：
   - PPO 需要 4 个模型同时驻留 GPU（策略、参考、RM、价值）
   - 这是 DPO 和 GRPO 诞生的主要动机

## 7. PPO 的优缺点

### 优点

| 优点 | 说明 |
|------|------|
| **简单** | PPO-Clip 仅约 20 行核心代码 |
| **稳定** | 裁剪机制天然防止策略崩溃 |
| **高效** | 每批次数据可重复使用 K 次（比 TRPO 更数据高效） |
| **通用** | 适用于离散和连续动作空间 |
| **成熟** | 被广泛验证，有大量调参经验 |

### 缺点

| 缺点 | 说明 |
|------|------|
| **On-policy** | 每步需要从当前策略采样，数据效率低于 off-policy 方法 |
| **Critic 开销** | 需要额外训练一个价值网络（与策略等大） |
| **超参数敏感** | ε、K、λ 等需要仔细调参 |
| **探索不足** | 熵正则化鼓励的探索有限 |
| **LLM 场景内存大** | RLHF 中需要 4 个模型驻留 GPU |

## 8. PPO vs 其他 RL 算法

| 算法 | 年份 | On/Off Policy | 是否需要 Critic | 复杂度 | 适用场景 |
|------|------|---------------|-----------------|--------|----------|
| **PPO** | 2017 | On-policy | 需要 | 中 | 通用，LLM RLHF |
| DQN | 2015 | Off-policy | 不需要（Q 函数代替） | 中 | 离散动作 |
| SAC | 2018 | Off-policy | 需要 | 高 | 连续控制 |
| TRPO | 2015 | On-policy | 需要 | 极高 | 高稳定需求 |
| A2C/A3C | 2016 | On-policy | 需要 | 低 | 简单任务 |
| **GRPO** | 2024 | On-policy | 不需要 | 低 | LLM 推理 RL |

## 9. 行业中的 PPO

### 9.1 标杆应用

| 应用 | 用途 |
|------|------|
| ChatGPT / InstructGPT | RLHF 的 RL 引擎 |
| OpenAI Five（Dota 2） | 击败世界冠军的多智能体训练 |
| DeepMind AlphaStar | StarCraft II 大师级 AI |
| Boston Dynamics 机器人 | 运动控制 |
| NVIDIA Isaac Gym | 大规模并行 RL 训练 |

### 9.2 现代改进

- **Phasic Policy Gradient（PPG）**：将策略和价值训练分离到两个阶段
- **Decoupled PPO**：解耦策略和价值网络的更新频率
- **PPO-EWMA**：使用指数加权移动平均来稳定更新
